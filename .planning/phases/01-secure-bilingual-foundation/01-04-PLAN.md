<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated user -> public schema | JWT claims and role settings determine allowed rows and operations. |
| app server -> database helpers | Server code depends on database-owned role checks for admin boundaries. |
| local migration -> generated client types | Schema changes become TypeScript contracts consumed by auth and shell code. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04-01 | Information Disclosure | `public.profiles` RLS | mitigate | Owner/admin SELECT policies plus pgTAP anon/customer A/customer B/admin matrix. |
| T-01-04-02 | Elevation of Privilege | `public.user_roles` | mitigate | No client write policies; admin authority stored in protected rows, not `user_metadata`. |
| T-01-04-03 | Tampering | `private.is_admin()` | mitigate | Private schema, revoked direct access, `security definer`, fixed `search_path`, and tests proving direct execution denial. |
| T-01-04-04 | Repudiation | role assignment metadata | accept | Phase 1 stores assignment metadata for boundary proof; full sensitive action audit is scoped to SEC-03 in Phase 4. |
| T-01-04-05 | Information Disclosure | generated types and Supabase clients | mitigate | Types contain schema shapes only; privileged credentials are not generated or placed in client code. |
</threat_model>

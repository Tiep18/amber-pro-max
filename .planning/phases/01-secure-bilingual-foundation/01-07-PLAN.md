<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cookie/session -> protected render | Auth cookies must be verified server-side before account/admin UI renders. |
| authenticated user -> admin shell | Customer-controlled requests may attempt direct admin access. |
| session-bound client -> RLS policies | Protected shell reads must stay within database-enforced row boundaries. |
| admin status -> rendered UI | Role state is sensitive and must not leak raw records or diagnostics. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-07-01 | Spoofing | `requireUser()` and `requireAdmin()` | mitigate | Use `auth.getClaims()` before protected rendering and avoid trusting cookie-only session reads. |
| T-01-07-02 | Elevation of Privilege | admin shell and actions | mitigate | Repeat `requireAdmin()` at server boundaries and rely on Plan 01-04 RLS/private role helper; never authorize from `user_metadata`. |
| T-01-07-03 | Information Disclosure | protected UI render | mitigate | Server-render guards decide before shell output; no client-side protected-content flash. |
| T-01-07-04 | Information Disclosure | account/admin pages | mitigate | Do not display raw roles, internal IDs, tokens, provider payloads, service keys, or authorization diagnostics. |
</threat_model>

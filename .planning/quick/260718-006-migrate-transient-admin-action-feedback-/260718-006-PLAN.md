# Quick Task 260718-006: Admin toast notification migration

## Goal

Replace transient admin action alerts with a consistent Sonner toast system without weakening accessibility, progressive form behavior, security boundaries, or durable operational feedback. Apply the reviewed external plan at `C:\Users\Pay Oo\Downloads\implementation_plan.md` against repository baseline `069d02e`.

## Scope guardrails

- Admin surfaces only; storefront/customer notification work is deferred.
- Toast only action completion and recoverable non-field failures. Keep field validation, publish blockers, readiness/load failures, sensitive tokens, VietQR evidence results, and media cleanup warnings inline.
- Use stable result-to-copy mappings. Never toast arbitrary exception text or unsanitized provider/server payloads.
- Fire interaction feedback inside the handler/reducer when possible. Use a guarded effect only where `useActionState` or redirect/query state is the sole result channel.
- Preserve pending states and confirmations. Toasts do not replace loading/disabled controls.
- Keep client boundaries narrow and serialized props minimal; do not turn full order/fulfillment display trees into Client Components.
- Preserve unrelated user work and do not change commerce/payment/fulfillment semantics.

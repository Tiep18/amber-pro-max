# Quick Task 260714-x6u: Fix admin catalog draft and publish flow

## Task 1 - Separate draft persistence validity from publish readiness

**Files**

- `supabase/migrations/20260714220000_allow_incomplete_catalog_drafts.sql`
- `supabase/tests/database/02_atomic_catalog_product_save.test.sql`
- `supabase/tests/database/02_catalog_model.test.sql`
- `src/catalog/schemas.ts`
- `src/catalog/actions.ts`
- `src/app/admin/catalog/catalog-data.ts`
- `src/types/supabase.ts`
- `tests/unit/catalog/publish-checks.test.ts`

**Action**

- Add one forward-only migration; do not edit historical catalog migrations. Make `product_translations.slug` nullable so multiple incomplete drafts do not collide on an empty localized slug, retain the non-empty slug format/locale uniqueness guarantees, and relax only the title non-blank and enabled-offer-price constraints that currently prevent intermediate draft state. Keep nonnegative integer money, currency/market pairing, locale, product type, JSON-object specifications, foreign keys, and RLS unchanged.
- Keep title stored as a non-null string but allow it to be blank for a draft. Normalize an empty/whitespace slug to `null` at the admin save boundary rather than persisting an empty sentinel. Permit `{enabled: true, priceMinor: null}` as draft input without weakening non-empty price validation.
- Replace the current public `catalog_publish_issues(uuid)` definition in the new migration without changing its signature or grants. Preserve every existing blocker, including the added `incompatible_product_data` check, and explicitly report a blank localized title as `missing_translation`, a null/blank slug as `missing_slug`, and an enabled offer without a price as `invalid_market_offer` (while retaining `missing_market_offer` when no complete enabled offer exists).
- In the same migration, drop and recreate the affected invariant triggers without renaming their public behavior: include `title` in `product_translations_preserve_publish_invariant_update`; add an `AFTER INSERT` market-offer trigger; and keep the market-offer update trigger watching `product_id`, `market_code`, `enabled`, and `price_minor` with a condition that runs whenever any of those eligibility facts changes. Route every trigger through `private.maintain_catalog_publish_invariant()` so it evaluates both old/new product ids where applicable. Preserve the existing delete triggers and all media, asset, variant, inventory, shipping, and product-type invariant coverage.
- Relax `productDraftSchema` only for publish-readiness omissions: empty localized title/slug and a missing price on an enabled offer are valid draft states. Continue rejecting malformed non-empty slugs, invalid UUIDs, negative/fractional prices, invalid specifications JSON, and invalid collection order values.
- Update the save payload to send null for an empty slug. Regenerate `src/types/supabase.ts`, update catalog loader/list row types for nullable slugs, and make untitled drafts render a stable `Untitled product` fallback without weakening published storefront assumptions.
- Extend pgTAP coverage to save and reload an incomplete aggregate atomically, prove a second incomplete product can also use null slugs, and prove publish remains blocked with the expected stable issue codes until required data is complete. Starting from a fully eligible published fixture, additionally prove (a) updating a localized title to blank demotes it to draft, (b) inserting an enabled null-price offer for a previously absent market demotes it, and (c) updating an existing offer to enabled with a null price demotes it. Add schema unit tests for permitted omissions and rejected malformed values.

**Verify**

- `npm run db:lint`
- `npm run db:test`
- `npm run db:types` and confirm `git diff --exit-code src/types/supabase.ts` after generation

**Done**

- Incomplete product work can be saved without placeholder content, while every D-17/D-18 publish requirement remains enforced by the database and public catalog eligibility is unchanged.

## Task 2 - Publish the current editor snapshot, never the stale saved snapshot

**Files**

- `src/catalog/actions.ts`
- `src/components/admin/catalog/product-form.tsx`
- `tests/unit/catalog/publish-checks.test.ts`

**Action**

- Add a save-then-publish Server Action that accepts the complete `ProductDraftInput` for an already-created product. Authorize before creating a database client, validate the structural draft schema and required product id, then save the aggregate through `admin_save_catalog_product` before invoking `publish_catalog_product` with the returned id.
- Share internal save/publish helpers rather than calling exported Server Actions from one another. Preserve monitored error boundaries, safe result shapes, catalog cache invalidation, and mapped `catalog_publish_issues` behavior. A validation/save error must return before any publish RPC; a successful save followed by blockers returns those blockers and leaves the current snapshot saved.

Keep publish disabled until the first save unlocks media/variant workflows. On blocked or published results, update `savedSignature` from the exact submitted snapshot so the UI accurately reports that the draft is saved; do not derive it from mutable state after awaiting the action. Preserve field navigation and blocker labels.
- Prevent double submissions using the existing transition state. Do not remove database publish checks or replace them with readiness indicators; the client readiness summary remains advisory only.
- Unit-test RPC order and payload identity, including an unsaved changed price/title reaching the save RPC before publish, save failure short-circuiting publish, blocker lookup after successful save, and the existing safe monitoring/no-raw-content guarantees.

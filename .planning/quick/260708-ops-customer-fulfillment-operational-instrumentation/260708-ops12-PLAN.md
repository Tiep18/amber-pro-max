# Customer Fulfillment Operational Instrumentation

## Goal

Record sanitized operational failures for customer-facing fulfillment access paths that currently return generic states without enough durable diagnostic context.

## Scope

- Account order history and pattern library query failures.
- Guest order reopen/claim email enqueue failures.
- Guest order claim mutation/audit failures, including Supabase responses that return an `error` object instead of throwing.
- Keep customer PII and secrets out of operational facts: no emails, raw claim tokens, signed URLs, or payload dumps.

## Plan

1. Add failing unit tests for recorder calls and safe facts across account queries and guest claim/reopen failures.
2. Instrument the fulfillment paths with `recordOperationalFailure`.
3. Ensure claim mutation returned errors no longer report `claimed`.
4. Run targeted fulfillment tests, typecheck, lint, update GSD summary/state, and commit scoped files only.

## Verification

- `npm run test:unit -- tests/unit/fulfillment/account-access.test.ts`
- `npm run typecheck`
- `npm run lint`

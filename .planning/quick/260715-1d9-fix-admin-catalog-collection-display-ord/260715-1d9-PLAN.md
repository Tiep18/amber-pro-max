# Quick Task 260715-1d9: Fix collection display order and catalog security boundary

## Task 1 - Assign collision-safe defaults without renumbering existing collection products

**Action**

- Add one forward-only migration defining `public.admin_catalog_collection_next_orders(target_collection_ids uuid[])`, returning exactly one `{collection_id, next_display_order}` row for each distinct requested collection id. Compute `coalesce(max(collection_products.display_order) + 1, 0)` inside PostgreSQL with a left join so empty collections return `0`; reject null/oversized request arrays, deduplicate repeated ids deterministically, and raise a stable invalid-argument error when any requested collection does not exist rather than returning a partial result. Keep results integer and nonnegative; do not compact gaps or mutate any membership.
- Make the aggregate RPC `security definer`, fix `search_path` to `public, private, pg_temp`, and fail with `42501 admin access required` unless `private.is_admin()` succeeds. Revoke execute from `PUBLIC`/`anon`, grant only `authenticated`, and retain table RLS/grants unchanged. Do not expose a generic service-role or public catalog aggregate.
- Extend only collection options with a typed `nextDisplayOrder`. In the admin loader, obtain localized collection options, pass their exact ids to the aggregate RPC, fail closed through `assertCatalogAdminQueryResults`, and join the small per-collection result back to the options. Keep categories, techniques, tags, and the collection-loading chain in the existing outer `Promise.all`; do not fetch raw `collection_products` rows into Next.js or the browser.
- Add a small pure catalog-ordering helper only for joining/reconciling the bounded RPC results. It must reject/make impossible a missing, negative, fractional, or non-finite aggregate instead of silently converting it to `0`; an empty collection's explicit RPC result is the only source of the zero default.
- Reconcile selector changes in `ProductForm` with this precedence: current draft membership order, initially loaded persisted membership order, then the option's `nextDisplayOrder`. This preserves explicit edits and remove-then-reselect behavior while assigning new memberships a collection-specific append position instead of the current hard-coded `0`.
- Keep the numeric input editable and keep `productDraftSchema` plus the database constraints authoritative for integer/nonnegative input. Do not renumber another product, use an upsert that can overwrite `(collection_id, display_order)`, or weaken `unique (collection_id, display_order)`. A race between concurrently loaded editors may return the existing sanitized save error; it must not overwrite an occupied order.
- Add pgTAP coverage for function existence/signature, security-definer/search-path/grants, non-admin rejection, deterministic duplicate-id handling, missing-id rejection, empty collections, sparse orders, and correct per-collection independence. Insert more than the usual PostgREST row ceiling (for example 1,205 memberships in one collection with `generate_series`) and prove the RPC returns `1205`, demonstrating that no client row truncation can corrupt the maximum.
- Unit-test RPC-result joining and preservation of current/persisted values, remove-then-reselect, deterministic new-selection defaults, and fail-closed malformed/missing aggregate data. Add an explicit schema regression that rejects negative/fractional collection order. Update the focused admin E2E flow to create or reuse an occupied collection order, select that collection for a different product without manually correcting the default, save, and assert the new membership receives the next order while the pre-existing row is unchanged. Keep fixtures isolated and cleaned up.
- Regenerate `src/types/supabase.ts` after the migration and consume its generated function return type rather than casting an untyped RPC payload.

# Quick Task 260715-0sw: Harden admin catalog media boundary

## Task 1 - Authorize upload routes before multipart parsing

**Files**

- `src/app/api/admin/catalog/media/product-image/route.ts`
- `src/app/api/admin/catalog/media/pattern-pdf/route.ts`
- `tests/unit/catalog/media-routes.test.ts`
- `tests/unit/catalog/media-actions.test.ts`

**Action**

- Import and invoke `requireAdmin` in both catalog media `POST` handlers before `request.formData()` or any file/body read. Keep cheap same-origin and declared `Content-Length` rejection ahead of body parsing, but do not let either route reach multipart parsing unless admin authorization succeeds. Let the existing redirect/forbidden control flow propagate rather than converting it into a public JSON oracle.
- Retain `requireAdmin` inside `uploadProductImageAction` and `uploadPatternPdfAction` so direct Server Action invocation remains protected. Do not pass a client-controlled authorization flag or weaken the current `user_roles`/RLS checks.
- Preserve both route request ceilings and all action-level validation: image MIME/extension mapping and 10 MiB ceiling; PDF MIME, `.pdf` suffix, and 50 MiB ceiling; product existence/type checks; stable 400/413/500 response mapping for authorized requests.
- Add route-level Vitest coverage with mocked authorization, parser, and action boundaries. Prove a denied caller never invokes `request.formData()` or the upload action for either endpoint; an authorized caller parses once and delegates once; malformed multipart and oversized declared bodies retain their existing status/code behavior. Add/retain focused action tests proving invalid MIME, suffix, and actual file size are rejected even when request headers pass.

**Verify**

- `npm run test:unit -- tests/unit/catalog/media-routes.test.ts tests/unit/catalog/media-actions.test.ts`
- `npx eslint src/app/api/admin/catalog/media/product-image/route.ts src/app/api/admin/catalog/media/pattern-pdf/route.ts tests/unit/catalog/media-routes.test.ts tests/unit/catalog/media-actions.test.ts`
- `npm run typecheck`

**Done**

- Unauthorized requests cannot force multipart allocation or file parsing, while authorized uploads retain all current boundary validation and both Server Actions remain independently admin-only.

## Task 2 - Make storage deletion outcomes explicit and metadata-safe

**Files**

- `src/catalog/media-actions.ts`
- `src/components/admin/catalog/media-manager.tsx`
- `tests/unit/catalog/media-actions.test.ts`

**Action**

- Replace the fire-and-forget storage removal helper with a checked result that inspects Supabase Storage's returned `error`. Report failures through `monitoredMediaFailure` with stable action/error codes and only `productId`/`referenceId`; never record `object_path`, filenames, raw storage messages, file bytes, or bucket credentials.
- For explicit `removeProductMediaAction` and `removePatternPdfAction`, remove the storage object first. If Storage fails, return `{status: 'error', code: 'remove_failed'}` before clearing social-image references, deleting `product_media`, deleting `product_digital_assets`, revalidating, or claiming success. After storage succeeds, perform the existing bounded metadata mutation; propagate database failures as errors. A retry must be able to converge metadata after an object was already removed, including provider "not found" behavior, without weakening product/media ownership predicates.
- Cover every non-explicit cleanup call as well: image/PDF upload rollback after association failure and old-PDF cleanup after replacement. A rollback cleanup failure keeps the original association error result and records a separate sanitized cleanup failure. A replacement whose new asset is already associated must not be mislabeled as an upload failure; return a typed success-with-cleanup-warning result, record the cleanup failure, and show a warning alert in `MediaManager` so the admin knows the new PDF is active but orphan cleanup needs attention. Normal success and destructive error copy/refresh behavior remains unchanged.
- Add unit tests for image removal and PDF removal when Storage rejects deletion, asserting database metadata mutations, cache invalidation, and success messages do not occur. Cover successful storage-first ordering, upload rollback cleanup failure, old-PDF replacement cleanup warning, operational-recorder failure resilience, and sanitization of private object paths/raw provider errors from both returned results and monitoring facts.

**Verify**

- `npm run test:unit -- tests/unit/catalog/media-actions.test.ts tests/unit/catalog/media-routes.test.ts`
- `node --test tests/security/catalog-boundaries.test.mjs` (record the existing unrelated generic `object_path` regex baseline if it remains; do not edit that test in this task)
- `npx eslint src/catalog/media-actions.ts src/components/admin/catalog/media-manager.tsx tests/unit/catalog/media-actions.test.ts`
- `npm run typecheck`
- `npm run build`

**Done**

- Storage failures are never silently discarded, explicit deletion does not remove metadata or announce success when the object remains, replacement cleanup warnings match the actual committed state, and no private path/provider detail crosses the admin response or monitoring boundary.

## Boundaries

- Do not change catalog collection ordering, optimistic concurrency, draft/publish behavior, variant pricing, checkout authority, or the known generic security-regex false positive.
- Do not make `pattern-pdfs` public, expose signed URLs from admin actions, introduce service-role credentials in application code, or loosen existing RLS/storage policies.
- Do not trust `Content-Length` as file validation; retain the action-level MIME/name/actual-byte-size checks after authorization.
- Do not rewrite historical migrations or add a database migration for this quick task. If post-storage database deletion fails, return an error and preserve bounded retry/monitoring behavior; do not claim cross-service atomicity.
